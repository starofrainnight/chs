通用C&C;++编程规范

Date Mon 30 May 2016 By starofrainnight Category 通用C&C;++编程规范

1 前言

原则上, 除了参考手册这类的文档型文件内容(仅内容,文件名仍然需要使用英文)可以使用中文进行编写, 其它的所有源代码/注释/文件名均使用英文进行编写( 禁止使用拼音进行描述 ). 此原则是为了避免在不同的源代码管理, 不同语言的开发人员, 不同的操作系统, 不同的编译器, 不同的编辑器环境下出现各种文字编码问题而设定, 请严格遵守.

2 源代码风格

2.1 命名约定

2.1.1 通用命名规则

  1. 所有命名尽可能表达其功能, 长没关系, 只要意思清楚, 例如: 采用 GUIShowContainerPictures 而不采用 GUIShowContPics. 除编译器对命名长度有特殊规定外, 均使用能清晰表达其意思的命名, 而不是缩写.
  2. 不要使用匈牙利命名法, 例如: m_hWnd 可以写作 m_WindowHandle, 尽量把意思表达清楚, m_iValue 直接写作 m_Value.
  3. 尽量用const代替#define 来创建常量
  4. 尽量用enum代替#define 来创建常量集合
  5. (c++) 尽量用内联(inline)函数代替#define宏
  6. (c++) 尽量用new和delete代替malloc 和 free
  7. (c++) 尽量用输入输出流(iostreams)代替stdio
  8. 命名修饰词
    1. on_xxx_xxx　当某事件发生
    2. is_xxx 是否判定
    3. has_xxx 有无判定
    4. get_xxx 获取某物, c语言必然加入get 前缀, c++ 则遵循以下规则
      1. 当 xxx 可作为动词使用时, 使用 “get_”前缀
      2. 当 xxx 的 get 操作是需要输入参数时,使用 “get_”前缀, 例如: get_char( char * c ); // 获取一个字符并存储到 c 中
      3. 除以上两种情况外,直接使用 xxx() 作为获取操作即可
      4. set_xxx 设置某物

2.1.2 注释

参照 3. 的注释规范.

2.1.3 文件名

  • 普通 C++ 类库文件名: aaa_bbb.h (全小写, 下划线分隔字)
  • GUI 框架相关文件名按 GUI 框架标准, 如 MFC: AaaBbb.h (里面的是类 CAaaBbb)
  • 普通 C 库文件名与 (1) 相同
  • 文件夹命名 aaa_bbb
  • 生成程序名与该文件夹或项目名相同

2.1.4 类型名

类型名采用全小写, 并以下划线分隔各字，最后以“_t”后缀结尾。例如：

class object_t 
{
	… 
};

enum mode_t 
{
	… 
};

typedef object_t new_object_t;

2.1.5 变量名

变量名全小写以下划线分隔字，根据所在域不同，有不同的命名前缀：

  • 当变量属于全局变量时，以“g_”为前缀
  • 当变量属于文件局部变量时，以“s_”为前缀
  • 当变量属于类内部成员变量时，以“m_”为前缀
  • 当变量属于函数内临时变量时，没有前缀。 例如： 
    //========================================== 
// test.hpp 内容:
// 全局变量 
extern int g_help_option; 

class test_t 
{ 
protected: 
	int m_value; ///< 类内部成员变量

public:
	test_t() 
		:m_value( 0 )
	{
		int ret = 12; ///< 函数内临时变量 
	} 
};

//========================================== 
// test.cpp 内容 : 

int g_help_option = 1;

static int s_total = 0; ///< 文件局部变量

2.1.6 常量名

常量名规则与变量名规则相同。

2.1.7 枚举值

枚举值采用前缀为类型名缩写, 全大写加下划线分隔字, 例如:

enum this_is_a_enum_t
{ 
	TIAE_THIS_IS_AN_VALUE,
	TIAE_THIS_IS_VALUE_TWO,
};

2.1.8 函数名

函数名采用全小写加下划线分隔字，例如：

int random_value_generate()
{
	return rand(); 
}

2.1.9 名字空间命名

名字空间采用全小写下划线分隔字的形式，例如：

namespace device 
{
	namespace sub_device 
	{
		...
	}
}

2.1.10 模板名

模板名命名规则与类型名相同。

模板的参数 typename,以大小写区分（驼峰式命名），Type 结尾, 例如:

template < typename VectorType > 
class vector_template_t 
{ 
	... 
};

2.1.11 宏名

宏名采用全大写以下划线分隔字，例如：

当以宏来模拟函数时，或实现函数替换时，可采用与函数命名规则，

#define THIS_IS_AN_MACRO 12
#define LONG_VALUE 0x1234 
#define MAKE_LONG( value ) ((long)(value))

2.1.12 标签名

标签名使用全大写, 前缀为 LABEL, 附带功能描述的方式命名, 如:

void test() 
{ 
	int value = 0;
	if( value ) 
	{ 
		goto LABEL_GET_NEXT_FILE_NAME; 
	}
	value = 3;

LABEL_GET_NEXT_FILE_NAME:
	return ; 
}

2.1.13 特殊规则

根据情况判断编写风格

第三方GUI库相关

当你所编写的类或函数属于第三方GUI库部分, 则必须使用与GUI相关的源码规范.

MFC

例如你正在写一个摄像头的PC软件, 使用的是MFC, 你写的是MFC的CButton控件的 派生类, 则使用如下规范:

  • 类名: CXXXXX (XXXX为大小写混合的英文字符串, 如: CBitmapDialog )
  • 类内成员变量: m_Xxxx ( 如: int m_TestVar )
  • 类内静态成员变量: s_Xxxxx ( 如: static int s_TestVar )
  • 类成员函数名: XxxxXxxx ( 如: FindDialogSettings )
  • 类内事件函数命名: OnXxxxXxxx ( 如: OnPresentationButton )
  • 类内控件变量命名: m_XxxxTTT ( 如: m_ChangeStateButton ) TTT 为控件类型, 类型命名如下:
    • Button : 所有 Button 类, 包括CButton, CBitmapButton ...
    • Static : 所有静态文本及相关类, 包括 CStaticText...
  • 类内各种变量都需要进入Protected模式, 使用Get 或 Set 前缀来编写其读写例程( 如 GetEnabled(), SetEnabled() )
  • 指针类型后加 Ptr 或根据情况而不加: CWindow * AWindowPtr 或 char * str 文件名: XxxxYyyy (大小写混合)
  • 所有的资源ID(控件ID,对话框ID,图片ID): TTT_SSS_XXX (全ID使用大写)
    TTT 为 资源ID标记, 会自动生成的, 如对话框的ID为 IDD_DIALOG2, IDD即为其资源ID标记
    XXX 为 资源类型, 一般控件类型会由IDE自动生成, 例如IDC_BUTTON1, BUTTON为其资源类型
    SSS为控件的名称, 控件名称使用下划线分隔字
    示例: 有一资源ID为 IDC_BUTTON1, 你只需要修改为 IDC_THIS_IS_A_TEST_BUTTON
Qt

当你编写的程序与Qt相关时，则采用类似Java风格的命名风格。

第三方GUI库无关C++规则

当你所编写的类或函数与GUI完全无关时, 并且你使用的是 C++, 则必须使用以下的源码规范.

  • 使用命名空间来规范名字
    所有的公司类均使用 xxxx (公司小写简称) 命名空间, 在 xxxx (公司小写简称) 命名空间之下是所有模块的命名空间, 例如你有一个函数属于GUI模块的, 名字为 set_display_width, 则其空间包含关系如下： xxxx::gui::set_display_width( ... )
  • 命名空间(namespace): 全小写, 下划线分隔
  • 类名: xxxx_yyy_t (全小写, 下划线分隔, 以_t为后缀, 如: class window_handle_t; )
  • 类内各种变量需要进入Protected模式, 使用get 或 set 前缀来编写其读写全程( 如 get_enabled(), set_enabled() 注意!!是小写的! )
  • 类型: aaaa_bbb_t (全小写, 下划线分隔, 以_t为后缀, 如: typedef vector<int> int_vector_t)
  • 函数名: xxx_xxx (全小写, 下划线分隔字, 如: void this_is_a_function(void) ) 指针类型后加 _ptr (小写!) 或根据情况而不加: AClass * class_ptr 或 char * str
  • 文件名: xxxx_yyyy (全小写, 下划线分隔) 模板参数: template < typename _allocator_type > (前置 “_”，后置 “_type”)
第三方GUI库无关C规则

当你所编写的类或函数与GUI完全无关时, 并且你使用的是 C, 则必须使用以下的源码规范.

  • 函数名: mmm_xxx_xxx (全小写, 下划线分隔字, mmm为模块名称, 如: void gui_set_dispaly_width( int width ) )
  • 属性获取函数同 2.2.2, 使用 mmm_get_xxx_xxx 形式
  • 代表一个对象类型时使用 xxxx_context_t, 当使用其作命名时可直接使用其前缀作为对象名, 因为此对象类型内的属性可唯一标识该对象.如 xxxx_context_t xxxx;
  • 其它与 2.2.2 类似
2.1.13.4. 多种编码风格的库共同使用

当源代码中存在多种风格的库使用时，我们以其中两种为主，其中一种作为一般性的编码风格，另一种则作为GUI部分的编码风格。

例如：我们项目中存在Qt作为GUI库，则一般的单元命名均以公司基本编码标准为准，仅在Qt窗口类函数中采用Qt的风格。而且仅针对类内成员及临时变量采用Qt的风格，其它的全局变量其它非GUI单元则采用公司的编码标准。

2.2 头文件

2.2.1 唯一性

为保证头文件的唯一性，头文件的防护宏以 “双下划线 + 文件名 + INCLUDED + GUID值” 的形式命名．例如：src/generator.h使用如下方式保护：

#ifndef __GENERATOR_H_INCLUDED_16165D58_F514_457F_9B2A_D97EA8C5ED07
#define __GENERATOR_H_INCLUDED_16165D58_F514_457F_9B2A_D97EA8C5ED07 
... 
#endif /* #ifndef __GENERATOR_H_INCLUDED_16165D58_F514_457F_9B2A_D97EA8C5ED07 */

采用“双下划线”前缀是为了避免在存在智能提示的IDE中与其它的标识符前缀冲突，提高智能提示的索引效率，实际上头文件的防护宏基本不作为常规用途，只用于保证头文件的唯一性。

GUID值则可保证此宏的唯一性，首先即使文件名相同，随机GUID值相同的概率低至几乎可忽略，假如GUID值是根据时间值生成的，那更可以保证此值的唯一性。

2.3 源代码

2.3.1 代码编写顺序

  1. 本地头文件引用 ( #include “…” )
  2. 系统头文件引用 ( #include <…> )
  3. 宏定义 ( #define … )
  4. 类型定义, 类型别名 ( class, struct, typedef )
  5. 函数前置声明
  6. 全局变量定义
  7. 函数体

2.4 源代码编写风格格式化

2.4.1. 使用软件AStyle对源代码自动格式化

使用 AStyle 配合以下的参数, 对源代码进行格式化:

--style=ansi --indent=force-tab --indent-cases --indent-namespaces --indent-preprocessor --max-instatement-indent=40 --min-conditional-indent=1 --break-blocks --pad-oper --pad-paren-in --break-closing-brackets --add-brackets --keep-one-line-blocks --keep-one-line-statements --align-pointer=middle

3 注释规范

3.1 文字要求

所有英文均需要按照英文的标准格式编写，句首字母要大写，例如：

Fixed program crash when remote terminal hang up. Because the audio encoder/ decoder be closed before audio threads stopped, they will access encoder or decoder during their lifetime. We now start audio threads after encoder / decoder.

3.2 Doxygen注释风格

所有注释均采用 Doxygen 风格, 尽量在注释中描述其功能及原理, 以便以后采用 Doxygen 直接生成相应的程序文档. 具体参考 Doxygen 注释使用介绍.txt 及例子 common.cpp和 common.h。 c 的注释只能使用 /* */

3.3 函数说明

3.4 文件头说明

3.5 对类, 结构, 枚举说明及其成员的说明

3.6 大行为说明

在进行编码时, 在某些大行为（例如进行某种复杂的操作过程）进行前, 进行一定的说明

3.7 描述文档链接

对一些复杂的算法进行描述时, 如果注释不足以描述, 则写入描述文档的链接

3.8 修复的说明

对一些BUG的修复, 在修复处进行足够的说明, 并提示后来人, 对此处的修改慎重

3.9 特殊称谓

  • 在 pthread 中
    // abc_thread_handle 被称作线程ID, 跟线程是绑定在一起的
    pthread_t abc_thread_id;
    // 线程句柄相对应的线程函数被称作 abc_thread
    void abc_thread( void * data );
  • 所有回调函数使用 _callback 后缀
    例如: rb_memory_stream_get_size_callback()
